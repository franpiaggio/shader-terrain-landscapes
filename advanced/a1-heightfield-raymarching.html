Heightfield raymarching

An SDF tells you the distance to the nearest surface in every direction. A landscape doesn't work that way — it's a height map, y = h(x, z). So we march the ray differently: step forward and ask "am I below the ground yet?"

Your win this lesson Write the heightfield march: walk the ray forward with growing steps and detect the moment it dips under the terrain — then render a raymarched mountain.

Why not sphere tracing?

Sphere tracing (core L4) needs a true distance field. A terrain height map gives you the surface height at a point, not the distance to it — so instead we use the classic heightfield march: advance along the ray and compare the ray's height to the terrain's height beneath it.

float castRay(vec3 ro, vec3 rd) {
    float t = 0.5, dt, lastD = ro.y - terrain(ro.xz);   // start above the ground
    for (int i = 0; i < 200; i++) {
        vec3 p = ro + rd * t;
        float d = p.y - terrain(p.xz);          // >0 above ground, <0 below
        if (d < 0.0)                             // just crossed the surface
            return t - dt + dt*lastD/(lastD - d); // interpolate the exact hit
        lastD = d;
        dt = 0.01*t + 0.05;                       // LOD: bigger steps farther away
        t += dt;
    }
    return -1.0;
}

Two ideas do the work: detect the sign change of p.y - terrain (you were above, now you're below → you crossed), and step proportionally to distance (dt = 0.01*t) so near ground is sampled finely and the far distance cheaply.1

Now you — complete the march, graded live

The marcher is missing its forward step, so the ray never moves and you only see sky. Add it. Check reads the pixels to confirm terrain appears.

Watch for Too-large steps tunnel through thin ridges (the ray jumps from above to far-below, missing a peak between). The interpolation softens it, but for sharp terrain you reduce the step factor or add a binary refine. Too-small steps just cost frames.
