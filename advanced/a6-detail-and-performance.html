Detail & performance

More octaves means crisper mountains — but the ray march samples the terrain hundreds of times per pixel, so every octave you add to it is paid hundreds of times over. The pro move is to spend detail only where it's seen: a cheap terrain for the march, an expensive one for the final shading. Then warp the ground so the ridgelines stop looking like a grid.

Your win this lesson Split the height map into a cheap marched terrain and a detailed normal terrain (level-of-detail), and add domain warping — sharper, more natural mountains that also run faster.

Where the time goes

Look at how often terrain() is called for a single pixel:

  • The march (castRay) — up to a couple hundred steps, one terrain lookup each. This dominates the cost.
  • The shadow ray — a few dozen more lookups.
  • The normal — exactly two lookups (a finite difference), once, at the hit.

So a fine octave added to the marched terrain is paid ~200×; the same octave added only to the normal is paid twice. The trick is obvious once you see it: march a coarse terrain, shade a detailed one.

Two terrains, one shape

Same function, different octave counts. The marcher and shadow use the cheap one; only the normal — the thing that actually carries surface detail to your eye — uses the expensive one:

float terrainM(vec2 x){ return H * fbmN(warp(x), 7);  }   // medium — march & shadows
float terrainH(vec2 x){ return H * fbmN(warp(x), 12); }   // high   — the normal only

// castRay(...)  uses terrainM      ← the hot path, kept cheap
// calcNormal(...) uses terrainH    ← runs twice, can afford the detail

Because the extra octaves are small, the coarse surface and the detailed surface share the same big shape — the ray lands in the right place against terrainM, and terrainH just adds crisp relief to the lighting. This is exactly how IQ's "Elevated" runs in real time.

Domain warping — kill the grid

Noise sits on an axis-aligned lattice, so ridgelines tend to run along x/z. Warp the input coordinate by a low-frequency noise before sampling and the ridges meander like real geology:

vec2 warp(vec2 x){ vec2 w = vec2(noise(x*0.05), noise(x*0.05 + 31.4));
                   return x + amount*(w - 0.5); }     // push the lookup sideways

Play — feel the trade-off

march detail sets the octaves the marcher uses — drag it down and the silhouette gets blocky and cheap, but the surface stays crisp because the normal always uses full detail. warp bends the ridgelines; sun moves the light. Camera flies forward.

Watch for Two ways this bites. If terrainM is too coarse, the marched silhouette and the detailed normal disagree near steep edges and you get shimmer — keep the medium terrain within a few octaves of the high one. And warp is sampled inside the hot march loop, so keep its noise cheap (one or two lookups), or you give back the speed the LOD just bought you.

Ship it: render at a fixed resolution

The biggest performance lever isn't in the shader at all — it's how many pixels you run it on. A raymarched terrain at full 4K is brutal; the same shader at a fixed 640² (then scaled up by the browser) is buttery and looks nearly identical in motion. The live demo does exactly this: a fixed-size canvas drawn with a single full-screen triangle. Resolution is your first dial when a landscape stutters.
